### Что сработало (топ-3 эффекта P14.3)

1. **completeness Δ −0.20 → +0.01** (P14.3.4 few-shot examples): RAG перестал быть «менее полным» чем no-RAG. Топ-5 примеров в prompts.yaml дали LLM эталон тона и структуры.

2. **historical_links 50% → 70%** (P14.3.1 product-token rerank): теперь и для `historical_request`, и для `product_query`/`smeta_request`/`bundle_query` — продукт из запроса промоутится наверх. Категория-mismatch (каталоги→баннеры, упаковка→открытки) почти полностью устранён.

3. **Client win +4pp** (P14.3.2 hybrid bypass): запросы вида «А почему дороже? Можете рассчитать?» теперь идут в `product_query` (regex_hybrid), а не блокируются как meta. Pricing pipeline отдаёт цифры.

### Что не сработало (известные регрессы / gaps)

1. **estimated_lead_time 0/400 fill** (P14.3.3): LLM не заполняет поле, хотя промпт-директива есть. Решение P14.4: переписать few-shot примеры с явной JSON-структурой (вместо «A_lead_time: ...» annotation), либо добавить post-processing fallback из таблицы стандартных сроков.

2. **print_flyer 35% (vs v1 50%)**: soft-rerank помог, но BGE-M3 в 1/3 случаев всё ещё ставит «листовки» выше «буклеты». Решение P14.4: добавить hard-filter для близких категорий в pricing-intents (но с graceful fallback).

## Recommendations v3 — P14.4 (следующая итерация)

### P14.4.1 — Fix estimated_lead_time fill rate (0% → ≥70%)

Текущая инжекция в prompts.yaml использует «A_lead_time:» формат — LLM не парсит это как JSON-ключ. Заменить на явный JSON-фрагмент с escaping:

```yaml
Пример 2 [дизайн логотипа → client]:
Q: «Сколько стоит разработать логотип для кафе?»
A: {{"summary": "Разработка логотипа для кафе — от 27 500 ₽...",
     "estimated_price": {{"value": 32500, "basis": "..."}},
     "price_band": {{"min": 25000, "max": 44000}},
     "estimated_lead_time": "3-5 рабочих дней"}}
```

Альтернатива (надёжнее): post-processing в `generator._scrub_structured_response`: если `estimated_lead_time` is None но есть `estimated_price` → fill из таблицы стандартных сроков по `summary`/category.

### P14.4.2 — Hard-filter для print_flyer (буклеты vs листовки)

В [retriever.py](RAG_RUNTIME/app/core/retriever.py) сейчас soft-rerank. Для категорий с известной BGE-M3 коллизией:

```python
CATEGORY_COLLISIONS = {
    "буклет": {"listovk", "флаер"},   # буклеты ≠ листовки
    "каталог": {"буклет", "брошюр"},   # каталоги ≠ буклеты
}
# Если query содержит «буклет» — drop docs с «листовк» в searchable_text
# (HARD filter, не soft).
```

### P14.4.3 — Discovery questions knowledge bank

Создать новый bridge_doc или knowledge_doc «Discovery questions per product»:
- Логотип: 5 готовых вопросов («какая ниша», «какая аудитория», «какие ассоциации»)
- Вывески: 5 вопросов («высота букв», «материал лицевой части», «тип подсветки»)
- Полиграфия: 5 вопросов («тираж», «бумага», «постпечать»)
- ...

Текущий ответ на «как мне выяснить параметры» — generic. Если будут конкретные вопросы в базе — LLM их процитирует.
